# 1. Product Overview & Goals

ToolRent is a cloud-based, multi-tenant SaaS platform purpose-built for tool and equipment rental businesses across India. It provides each rental business (tenant) with a fully branded, self-contained digital storefront and back-office management suite — eliminating dependency on spreadsheets, manual bookings, and fragmented cash workflows.

**Vision:** "Empower every tool rental business in India — from a single-shop owner in Pune to a multi-city fleet operator in Bengaluru — with enterprise-grade digital infrastructure, accessible from any device, at an affordable SaaS price."

**Mission:** To digitize the fragmented and largely offline tool rental market in India by providing rental businesses with an easy-to-use SaaS platform covering inventory management, customer bookings, payments, and communication — reducing operational overhead by up to 60%.

### Core Goals (MVP)
- Enable rental businesses to onboard and start accepting bookings within 24 hours
- Provide customers a seamless online tool browsing and rental experience (mobile-first)
- Support end-to-end rental lifecycle: browse → book → pay → track → return → review
- Handle security deposits and refunds via Razorpay (UPI-first)
- Deliver a scalable multi-tenant foundation supporting 100+ tenants at MVP launch

### Success Metrics

| Metric | Month 3 | Month 6 |
|---|---|---|
| Active Tenants | 25+ | 100+ |
| Monthly Bookings | 500+ | 5,000+ |
| Payment Success Rate | >95% | >97% |
| Customer CSAT | >4.0/5 | >4.3/5 |
| Platform Uptime | 99.5% | 99.9% |
| Avg. Onboarding Time | <2 hours | <1 hour |

# 2. Target Users & Use Cases

### Persona 1: Tenant Admin (Rental Business Owner)
- **Who:** Small to mid-size tool rental shop owner (construction equipment, power tools, party supplies)
- **Location:** Tier-1 and Tier-2 Indian cities (Mumbai, Pune, Bengaluru, Hyderabad, Jaipur, Lucknow)
- **Pain Points:** Manual booking logs, cash-only payments, no inventory visibility, high no-show rates
- **Goal:** Reduce admin overhead, accept digital payments, grow customer base online

### Persona 2: Staff / Manager
- **Who:** Employee managing day-to-day operations
- **Need:** Simple dashboard with clear task queues and booking status views

### Persona 3: Customer (End Renter)
- **Who:** Individual or contractor needing equipment short-term
- **Behavior:** Mobile-first, searches on Google, prefers UPI payment
- **Need:** Easy tool discovery, transparent pricing, instant booking confirmation, chat with owner

### Persona 4: Platform Super Admin (ToolRent Team)
- **Who:** ToolRent internal team managing the platform
- **Need:** Global admin panel with cross-tenant visibility, tenant approval, plan management

### Key Use Cases
1. Rental business registers and onboards; configures branding, tools, and pricing
2. Customer visits tenant's storefront, browses tools with availability calendar
3. Customer books a tool, pays deposit + rental fee via Razorpay UPI
4. Tenant admin receives booking notification, confirms or modifies booking
5. Customer and owner communicate via in-app chat for pickup/delivery coordination
6. Tool is returned; security deposit is released or partially withheld
7. Customer leaves a star rating and review; tenant can respond
8. Tenant views revenue reports, inventory utilization, and analytics on dashboard

# 4. Detailed System Architecture

ToolRent follows a layered, modular architecture with clear separation between presentation, API, business logic, and data layers. The system uses a shared database with tenant_id isolation strategy for MVP.

```
[ Browser / Mobile App ]
       | HTTPS
[ React SPA + PWA ] <-- Nginx / CDN (Cloudflare)
       | REST API / WebSocket
[ Spring Boot API Gateway / BFF ]
       |
[ Modular Spring Boot Services ]
  - Auth Service       - Tenant Service
  - Inventory Service  - Booking Service
  - Payment Service    - Chat Service
  - Review Service     - Notification Service
       |
[ PostgreSQL (Multi-Tenant, Shared DB) ]
       |
[ AWS S3 / Cloudinary ]  [ Razorpay ]  [ Redis Cache ]
```

### Architecture Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Multi-tenancy Model | Shared DB + tenant_id | Faster MVP, lower cost; migrate to schema isolation at scale |
| API Style | RESTful + WebSocket | REST for CRUD; WebSocket for real-time chat |
| Auth Strategy | JWT + Refresh Tokens | Stateless, scalable; short-lived access tokens |
| File Storage | Cloudinary + S3 | Cloudinary for image optimization; S3 for bulk storage |
| Cache Layer | Redis | Session caching, availability checks, rate limiting |
| Frontend | React SPA (PWA) | Mobile-first PWA for offline support and app-like UX |
| Payments | Razorpay | India-first, strong UPI/QR support, webhook-ready |

# 9. Multi-Tenant Architecture Strategy

### Tenancy Model Comparison
| Model | Isolation | Complexity | Cost | When to Use |
|---|---|---|---|---|
| Shared DB + tenant_id (MVP) | Row-level | Low | Low | 0–500 tenants |
| Schema-per-Tenant | Schema-level | Medium | Medium | 500–5,000 tenants |
| DB-per-Tenant | Full | High | High | 5,000+ tenants |

### Tenant Isolation Mechanisms

**A. Tenant Resolution**
- JWT Token: tenant_id embedded in JWT claims (primary method)
- Subdomain: `{tenant}.toolrent.in` parsed at Nginx level
- TenantContext (ThreadLocal): Spring filter stores tenant_id per request

**B. Data Isolation**
- All JPA queries include tenantId parameter
- Hibernate @Filter as secondary safety net
- PostgreSQL Row Level Security (RLS) as database-level backstop

**C. Subdomain Routing**
- Wildcard DNS: `*.toolrent.in` → Nginx
- Nginx extracts subdomain, passes as `X-Tenant-Subdomain` header
- API resolves tenant_id from subdomain via Redis-cached lookup

# 10. Security Considerations

### Authentication & Authorization
- JWT Access Tokens: 15-minute expiry, RS256 signed
- Refresh Tokens: 30-day expiry, stored hashed in DB, rotated on each use
- RBAC: super_admin > tenant_admin > tenant_staff > customer
- Tenant boundary check on every service method

### Threat Mitigation
| Threat | Mitigation |
|---|---|
| Cross-Tenant Data Leakage | tenant_id on every query + RLS + integration tests |
| SQL Injection | JPA parameterized queries; no raw SQL |
| IDOR | Resource ownership check before every operation |
| Brute Force Login | Rate limiting: 5 attempts / 15 min via Redis |
| Webhook Tampering | HMAC-SHA256 signature verification (Razorpay) |
| XSS | React output escaping + CSP headers via Nginx |
| Data in Transit | TLS 1.3 enforced + HSTS |
| Passwords | BCrypt cost factor 12; card data never stored |
| File Upload Abuse | MIME validation, 10MB limit, Cloudinary scanning |

### Additional Measures
- Jakarta Bean Validation on all DTOs
- Audit log for all write operations
- Secrets via AWS Secrets Manager (never in code)
- OWASP dependency scanning in CI pipeline
- Customer data deletion API (GDPR-lite compliance)

# 14. Risk Analysis & Mitigation

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Razorpay downtime | Low | High | Idempotent order IDs, webhook retry queue |
| Cross-tenant data leak | Low | Critical | RLS, integration tests, code review checklist |
| Scope creep | High | High | Strict P0/P1 triage, weekly sprint reviews |
| Low tenant adoption | Medium | High | Free tier, pilot program, onboarding concierge |
| DB performance degradation | Medium | Medium | Index optimization, Redis caching, query profiling |
| WebSocket scalability | Low | Medium | Redis pub/sub for multi-instance WebSocket |
| Poor mobile connectivity | High | Medium | PWA offline support, aggressive retry logic |
| Razorpay KYC compliance | Medium | High | Start compliance in Week 1, use Sandbox until approved |
| Developer availability | Medium | High | Buffer in Weeks 9–10; modular parallel development |
